# Aeolus — Project Planning Document

> **P2P Agent Negotiation Layer for the Mistral Worldwide Hackathon**

## 1. Project Summary

Build **Aeolus** — a decentralized peer-to-peer semantic negotiation layer that lets autonomous AI agents discover each other, negotiate task terms, and route work across a heterogeneous mesh of Ministral models — all without a central server. The project targets the **"Best Use of Agent Skills"** special award.

## 5. Negotiation Protocol Specification

### 5.1 Agent Capability Document (ACD)

```json
{
  "peer_id": "12D3KooW...",
  "name": "agent-alpha",
  "model_tier": "3B",
  "model_name": "ministral:3b",
  "capabilities": [
    "text summarisation",
    "sentiment analysis",
    "question answering"
  ],
  "capability_description": "I am a lightweight agent specialised in fast text processing tasks including summarisation, sentiment analysis, and extractive QA.",
  "status": "available",
  "max_concurrent_tasks": 2,
  "current_load": 0,
  "supported_protocols": ["aeolus/negotiate/v1"],
  "timestamp": "2026-02-28T10:00:00Z"
}
```

### 5.2 Negotiation Message Types

```
REQUEST  →  { type: "REQUEST", task_id, from, task_description, requirements: { max_latency_ms, min_quality, priority } }
OFFER    ←  { type: "OFFER", task_id, from, match_score, estimated_latency_ms, confidence, terms: {...} }
COUNTER  →  { type: "COUNTER", task_id, from, adjusted_terms: {...}, reason }
ACCEPT   →  { type: "ACCEPT", task_id, from, binding_terms: {...} }
BIND     ←  { type: "BIND", task_id, from, confirmation, execution_start }
RESULT   ←  { type: "RESULT", task_id, from, output, metrics: { actual_latency_ms, tokens_used } }
CANCEL   →  { type: "CANCEL", task_id, from, reason }
```

### 5.3 Negotiation Flow

```
Agent A (requester)          Network              Agent B (provider)
       |                        |                        |
       |-- REQUEST ------------>|-- (GossipSub) -------->|
       |                        |                        |-- LLM assesses fit
       |                        |<---- OFFER ------------|
       |-- COUNTER (optional) ->|----------------------->|
       |                        |<---- OFFER (revised) --|
       |-- ACCEPT ------------->|----------------------->|
       |                        |<---- BIND -------------|
       |                        |     ... execution ...  |
       |                        |<---- RESULT -----------|
```

## 6. Key Technical Decisions

| Decision | Choice | Rationale | Fallback |
|----------|--------|-----------|----------|
| P2P transport | py-libp2p | Real P2P, mDNS, GossipSub | NATS (single broker, simpler) |
| Local LLM runtime | Ollama | One-command install, OpenAI-compatible API | llama.cpp server |
| Model | Ministral 3B Q4_K_M | Fits in RAM, fast, function calling | Ministral 8B (if GPU available) |
| Embedding model | `nomic-embed-text` via Ollama | Small, good quality, same runtime | Mistral API embeddings |
| Message format | JSON | Debuggable, LLM-native | Protobuf (if perf needed) |
| Dashboard | Streamlit | Fastest to build, good enough for demo | FastAPI + HTMX |
| Identity | libp2p PeerID (Ed25519) | Crypto identity, no registry | UUID (simpler, less impressive) |

## 7. Risk Register & Mitigations

| # | Risk | Likelihood | Impact | Mitigation |
|---|------|-----------|--------|-----------|
| R1 | py-libp2p is too unstable / missing features | High | High | **Fallback to NATS** — pre-tested in spike S5. Loses "true P2P" but keeps the demo working |
| R2 | mDNS doesn't work on hackathon WiFi (client isolation) | Medium | High | Bring a **portable router** for a private subnet; or fallback to hardcoded peer addresses |
| R3 | Ministral 3B negotiation quality is poor (echoing problem) | Medium | Medium | Use **strong system prompts** with explicit strategic instructions + deterministic rules for hard constraints |
| R4 | LLM generates malformed JSON in negotiation | Medium | Medium | Use **JSON mode with schema enforcement** + Pydantic validation + retry logic (max 3 retries) |
| R5 | Ollama inference too slow on team hardware | Low | High | Pre-quantise to Q4_K_M; if still slow, use **Mistral API** as primary and local as secondary |
| R6 | Team member unavailable | Medium | High | All code in shared repo; each layer has a README; pair program critical paths |
| R7 | Demo fails live | Medium | Critical | **Pre-record a backup video**; have a scripted fallback demo; rehearse 3+ times |
| R8 | Scope creep into stretch goals | High | Medium | **Hard time-box**: no stretch goals until Phase 3 exits green. MVP first, always. |

## 9. Demo Scenario (Scripted)

### Setup
- **Agent Alpha** (Laptop 1): Ministral 3B — specialises in text summarisation & sentiment analysis
- **Agent Beta** (Laptop 2): Ministral 3B — specialises in translation (EN↔FR) & text rewriting
- **Agent Gamma** (Laptop 3 or same machine, different port): Ministral 8B — specialises in code review & complex reasoning

### Demo Script (~5 minutes)

1. **Discovery** (30s): Start all three agents. Dashboard shows them appearing one-by-one on the network graph. Agent Capability Documents populate the sidebar.

2. **Simple Task** (60s): Submit "Summarise this 500-word article about renewable energy" to Agent Beta (the translator). Beta receives it, assesses low match score for summarisation, broadcasts the request. Alpha offers (high match), Beta forwards. Alpha executes and returns the summary. Dashboard shows the full negotiation flow.

3. **Negotiation with Counter** (90s): Submit "Translate this technical document from English to French, under 2 seconds latency". Beta offers but with 3s estimated latency. Requester counters with "accept 3s if quality score ≥ 0.8". Beta accepts with quality guarantee. Task executes. Dashboard shows the COUNTER round.

4. **Tier Escalation** (60s): Submit "Review this Python function for security vulnerabilities and suggest fixes". Alpha and Beta assess low confidence. Gamma (8B) offers with high confidence. Task auto-routes to Gamma. Dashboard highlights the escalation.

5. **Resilience (stretch)** (60s): Kill Agent Alpha mid-network. Dashboard shows it disappearing. Submit another summarisation task — network re-routes to best available agent.

## 10. Pitch Structure (3–5 minutes)

1. **The Problem** (30s): "A2A and MCP solved agent communication — but only in client-server. What happens when agents are equals? Today, there is no protocol for that."

2. **The Gap** (30s): Show the protocol landscape diagram. Highlight the empty quadrant: decentralised + semantic negotiation.

3. **Aeolus** (30s): "A peer-to-peer semantic negotiation layer where agents discover each other, negotiate task terms, and route work — with zero central coordination."

4. **Live Demo** (2–3 min): Run the scripted scenario. Let judges see real agents on real machines finding each other and negotiating.

5. **Architecture** (30s): Flash the 3-layer diagram. Mention A2A/MCP compatibility. Mention Ministral 3B running locally.

6. **Future** (30s): "Aeolus is Layer 9 — the semantic negotiation layer the ecosystem is missing. Next: DID-based identity, on-chain SLA enforcement, cross-protocol bridge to A2A."
